# chrome-mcp-debug 技术方案设计文档

## 一、背景与目标

### 问题描述
在前端开发与调试过程中，AI agent 无法直接感知浏览器中发生的控制台报错（`console.error`、CSP 违规、资源加载失败等），开发者需要手动复制错误信息再提供给 agent，效率低下。

### 解决目标
构建一个 MCP（Model Context Protocol）服务，让 Claude Code、Cursor 等 AI agent 可以直接连接到正在运行的 Chrome 浏览器，实时读取控制台日志，实现自主调试前端问题。

---

## 二、整体架构

```
AI Agent（Claude Code / Cursor 等）
         │
         │  MCP stdio 传输协议（JSON-RPC）
         ▼
   MCP Server（index.ts）
         │
         │  chrome-remote-interface（HTTP + WebSocket）
         ▼
   Chrome 浏览器（--remote-debugging-port=9222）
         │
         │  Chrome DevTools Protocol（CDP）
         ▼
   浏览器标签页（Runtime domain + Log domain）
```

**核心设计原则：**
- 附着到已有 Chrome 实例，不主动启动浏览器
- 进程内共享状态（日志缓冲、CDP 连接），所有工具调用共用同一个 `ChromeManager`
- stdio 传输：stdout 完全保留给 MCP 协议通信，所有调试输出走 stderr

---

## 三、技术选型

| 技术 | 选型 | 理由 |
|------|------|------|
| MCP SDK | `@modelcontextprotocol/sdk` | 官方 TypeScript SDK，工具注册简洁 |
| CDP 客户端 | `chrome-remote-interface` | 轻量级，直接附着到已有 Chrome，不依赖 puppeteer |
| 参数校验 | `zod` | MCP SDK 内置集成，类型安全 |
| 运行时 | `tsx` | 无需编译直接运行 TypeScript，开发/部署均方便 |
| 模块系统 | ESM（`"type": "module"`） | MCP SDK v1.x 要求 ESM |

## 五、核心模块设计

### 5.1 类型定义（types.ts）

```typescript
type LogLevel = "log" | "warn" | "error" | "info" | "debug";

interface LogEntry {
  level: LogLevel;
  text: string;
  url?: string;         // 日志来源 URL
  lineNumber?: number;  // 来源行号
  timestamp: number;    // Unix 毫秒时间戳
}

interface PageInfo {
  id: string;    // CDP target ID，用于后续操作
  url: string;
  title: string;
  type: string;  // 通常为 "page"
}
```

### 5.2 ChromeManager 类（chrome.ts）

**私有状态：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `logBuffers` | `Map<targetId, LogEntry[]>` | 每个页面独立的日志环形缓冲，最多 500 条 |
| `clients` | `Map<targetId, CDP.Client>` | 已建立的 CDP WebSocket 连接 |
| `host` / `port` | string / number | Chrome 调试接口地址，默认 `localhost:9222` |

**方法列表：**

| 方法 | 说明 |
|------|------|
| `listPages()` | 调用 `CDP.List()` 获取所有 `type === "page"` 的 target |
| `attachToPage(id)` | 建立 CDP 连接，启用 Runtime + Log domain，注册事件监听 |
| `detachFromPage(id)` | 关闭 CDP 连接，保留日志缓冲供后续读取 |
| `getConsoleLogs(id, level, limit)` | 返回缓冲副本，支持按级别过滤，取最近 N 条 |
| `clearConsoleLogs(id)` | 重置缓冲为空数组 |
| `evaluate(id, expr)` | 通过 `Runtime.evaluate` 在页面上下文执行 JS |
| `pushLog(id, entry)` | 内部方法：追加日志，超出 500 条时从头部淘汰 |

**双事件源设计（关键）：**

```
Runtime.consoleAPICalled  ←── 捕获页面 JS 调用的 console.log/warn/error 等
Log.entryAdded            ←── 捕获浏览器级别日志：CSP 违规、资源加载失败、安全警告等
```

两者互补，`Log.entryAdded` 能捕获到不经过 console API 的系统级错误。

### 5.3 MCP 工具注册（index.ts）

服务启动时创建唯一的 `ChromeManager` 实例，所有工具共享：

```typescript
const manager = new ChromeManager();
const server = new McpServer({ name: "chrome-mcp-debug", version: "1.0.0" });
// 注册工具...
await server.connect(new StdioServerTransport());
```

## 六、MCP 工具规格

| 工具名 | 参数 | 返回说明 |
|--------|------|---------|
| `list_pages` | 无 | 所有页面标签的 id / url / title 列表 |
| `attach_to_page` | `targetId: string` | 成功/失败文本 |
| `detach_from_page` | `targetId: string` | 成功文本 |
| `get_console_logs` | `targetId`, `level`（默认 all）, `limit`（默认 100，最大 500）| 格式化日志文本，含时间戳、级别、来源位置 |
| `clear_console_logs` | `targetId: string` | 成功文本 |
| `evaluate` | `targetId`, `expression: string` | JSON 序列化的执行结果，或错误信息 |

日志输出格式示例：
```
[14:23:01.456] [ERROR] TypeError: Cannot read properties of undefined (reading 'map')
    (http://localhost:3000/app.js:142)
```

## 七、关键约束与注意事项

### ESM 导入路径
NodeNext 模块解析要求本地导入必须带 `.js` 后缀（即使文件是 `.ts`）：
```typescript
import { ChromeManager } from "./chrome.js";  // ✅
import { ChromeManager } from "./chrome";      // ❌ 运行时报错
```

### stdout 纯净性
MCP 通过 stdio 传输 JSON-RPC 消息，任何写入 stdout 的非协议内容都会破坏通信。调试输出必须走 stderr：
```typescript
process.stderr.write("debug info\n");  // ✅
console.log("debug info");             // ❌ 污染 MCP 通道
```

### 日志缓冲淘汰方向
保留最新的 500 条（从头部淘汰旧条目），因为 agent 通常关心最近的错误：
```typescript
buffer.splice(0, buffer.length - MAX_BUFFER);
```

### Chrome 连接前置条件
Chrome 必须以 `--remote-debugging-port=9222` 启动，且同一台机器上只能有一个 Chrome 实例监听该端口。
