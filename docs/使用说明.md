# chrome-mcp-debug 使用说明

## 一、前置条件

- Node.js 18+
- Google Chrome 浏览器
- Claude Code、Cursor 或其他支持 MCP 协议的 AI agent

---

## 二、安装

```bash
cd C:/Users/hew-f/work/plan/chrome-mcp-debug
npm install
```

## 三、启动 Chrome（必须）

每次使用前，需要以**远程调试模式**启动 Chrome。

**Windows（PowerShell）：**
```powershell
& "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --user-data-dir=C:\temp\chrome-debug-profile
```

**Windows（CMD）：**
```cmd
"C:\Program Files\Google\Chrome\Application\chrome.exe" ^
  --remote-debugging-port=9222 ^
  --user-data-dir=C:\temp\chrome-debug-profile
```

**说明：**
- `--remote-debugging-port=9222`：开启 CDP 调试端口
- `--user-data-dir`：使用独立的用户数据目录，避免与正常 Chrome 配置冲突（目录不存在会自动创建）

**验证 Chrome 是否就绪：**

在浏览器中访问 `http://localhost:9222/json/version`，若返回 JSON 信息则表示成功。

> **注意：** 如果你已经有一个普通模式运行的 Chrome，需要先关闭它，再以调试模式启动。同一端口只能有一个实例。

## 四、配置 MCP（以 Claude Code 为例）

### 方式一：全局配置（推荐）

编辑 `~/.claude.json`（Windows 下为 `C:\Users\hew-f\.claude.json`），添加：

```json
{
  "mcpServers": {
    "chrome-debug": {
      "command": "C:/Users/hew-f/work/plan/chrome-mcp-debug/node_modules/.bin/tsx",
      "args": ["C:/Users/hew-f/work/plan/chrome-mcp-debug/src/index.ts"]
    }
  }
}
```

### 方式二：项目级配置

在你的项目根目录创建 `.mcp.json`：

```json
{
  "mcpServers": {
    "chrome-debug": {
      "command": "C:/Users/hew-f/work/plan/chrome-mcp-debug/node_modules/.bin/tsx",
      "args": ["C:/Users/hew-f/work/plan/chrome-mcp-debug/src/index.ts"]
    }
  }
}
```

配置完成后**重启 Claude Code** 使配置生效。

### Cursor 配置

编辑 `~/.cursor/mcp.json`，格式与上方相同。

## 五、工具使用参考

配置成功后，agent 可使用以下 6 个工具：

### `list_pages` — 列出所有标签页
无需参数，返回当前 Chrome 中所有页面的 id、标题、URL。

```
> 使用 chrome-debug 列出当前 Chrome 中打开的所有标签页
```

返回示例：
```
[1] id: A1B2C3D4E5F6...
    标题: 我的应用 - 首页
    URL: http://localhost:3000

[2] id: F6E5D4C3B2A1...
    标题: GitHub
    URL: https://github.com
```

---

### `attach_to_page` — 开始监听控制台

```
> 连接到 id 为 A1B2C3D4E5F6 的页面，开始捕获控制台日志
```

参数：
- `targetId`：来自 `list_pages` 的页面 id

> **注意：** 只有调用此工具后，后续的控制台日志才会被捕获。建议在复现问题之前先调用。

---

### `get_console_logs` — 读取控制台日志

```
> 获取该页面所有 error 级别的控制台日志，最多 50 条
```

参数：
- `targetId`：页面 id
- `level`：日志级别，可选值 `all` / `log` / `info` / `warn` / `error` / `debug`，默认 `all`
- `limit`：返回最近 N 条，默认 100，最大 500

返回示例：
```
[14:23:01.456] [ERROR] TypeError: Cannot read properties of undefined (reading 'map')
    (http://localhost:3000/src/components/List.jsx:42)
[14:23:05.120] [WARN] Each child in a list should have a unique "key" prop.
    (http://localhost:3000/src/components/List.jsx:15)
```

---

### `clear_console_logs` — 清空日志缓冲

```
> 清空该页面已捕获的日志
```

用于在重新复现问题前清理历史日志，避免干扰。

---

### `detach_from_page` — 停止监听

```
> 断开与该页面的连接
```

停止捕获新日志，已有日志仍可通过 `get_console_logs` 读取。

---

### `evaluate` — 在页面中执行 JavaScript

```
> 在该页面执行：document.querySelectorAll('.error-item').length
```

参数：
- `targetId`：页面 id
- `expression`：任意 JavaScript 表达式

适合查询页面状态、读取变量值、手动触发函数等调试操作。

---

## 六、典型使用流程

以下是一个完整的调试会话示例：

```
1. 用户：我的页面有报错，帮我看看

2. Agent 调用 list_pages
   → 找到 http://localhost:3000，id 为 ABC123

3. Agent 调用 attach_to_page(targetId="ABC123")
   → 开始监听

4. 用户：（在页面上触发出错的操作）

5. Agent 调用 get_console_logs(targetId="ABC123", level="error")
   → [14:25:10.321] [ERROR] Uncaught TypeError: map is not a function
       (http://localhost:3000/src/utils.js:18)

6. Agent 调用 evaluate(targetId="ABC123", expression="typeof window.myData")
   → "undefined"

7. Agent 定位到问题：myData 未初始化就调用了 .map()，给出修复建议
```

## 八、常见问题

**Q：调用 `list_pages` 提示连接失败？**
A：确认 Chrome 是以 `--remote-debugging-port=9222` 启动的，访问 `http://localhost:9222/json/list` 验证。

**Q：attach 成功但 `get_console_logs` 返回空？**
A：attach 之后才开始捕获，attach 之前的日志不会被记录。请在 attach 后重新触发页面操作。

**Q：`evaluate` 返回 `undefined`？**
A：确认表达式有返回值，例如用 `window.myVar` 而不是 `var x = 1`（赋值语句无返回值）。

**Q：Chrome 关闭后重新打开，需要重新配置吗？**
A：不需要重新配置 MCP，但需要重新调用 `attach_to_page`（每次 Chrome 重启后 targetId 会变化，先调用 `list_pages` 获取新 id）。
